# 构建大规模、可信任、有状态的Trading Agent Researcher
## Harness Engineer

### 1. 什么是 Harness Engineer

`Harness Engineer` 不是单纯做 Prompt 的人，也不是只负责接模型 API 的人。这个角色的核心职责，是把大模型、Agent、工具链、业务系统和人的协作方式组织成一个可运行、可复用、可观测的执行底座。

如果把模型理解成“大脑”，那么 harness 就是“大脑工作的工程环境”。它决定模型在什么上下文里思考、能调用哪些工具、如何与外部系统交互、如何被约束、如何被审计，以及如何在一个真实业务流程里持续产出稳定结果。

因此，`Harness Engineer` 关注的重点通常不是“让模型多说一点”，而是下面这些工程问题：

- 如何定义 Agent 的运行边界，而不是让它无限制地直接操作系统。
- 如何组装上下文，让模型拿到刚好够用、且不会失真的信息。
- 如何把工具调用、任务派发、权限控制、状态流转串成稳定流程。
- 如何让一次执行可复盘、可追踪、可重放，而不是只留下几段聊天记录。
- 如何把通用模型能力转译成某个业务域里的实际生产力。

从这个意义上说，`Harness Engineer` 更接近“Agent 运行时工程师”或“模型执行环境工程师”。他关心的是运行机制，而不是单点技巧。

#### 2. 我们的产品底座：量化交易系统

我们的产品底座不是一个通用聊天产品，也不是一个纯粹的 Agent Playground，而是一个以量化交易系统为核心场景的业务系统。

这件事很重要，因为它决定了 harness 的设计目标不是“通用性优先”，而是“在量化研究与交易链路中可用”。

在量化交易场景里，Agent 的价值不来自开放式表达，而来自对研究流程的贴合。一个有效的 harness，必须服务于这些真实问题：

- 因子研究是否遵循既定方法论，而不是随意生成一个看起来合理的结论。
- 回测、验证、审计、复盘是否有明确分工，而不是把所有推理压给一个 Agent。
- 数据质量、交易成本、风控约束、市场环境这些关键变量，是否在上下文中被显式纳入。
- 研究结论是否能进入团队协作链路，而不是停留在单个模型的一次回答。
- 每个角色的输出是否带有责任边界，例如谁负责构想、谁负责反驳、谁负责裁决、谁负责工程落地。

也就是说，量化交易系统不是 Harness 上面随便接的一个业务插件，而是决定 Harness 结构的“底座约束”。

在这个前提下，Harness Engineer 做的不是一个抽象的 Agent Sandbox，而是一个面向量化研究、评审、回测、风险讨论和执行协同的运行时。

globalShopping 3. 我们项目中的 Harness 实践

在这个项目里，`agentGroup` 扮演的是 Harness 的角色。它不是直接承载量化业务规则本身的研究仓库，而是作为 Agent 的运行时宿主，去组织 Persona、Session、Topology、Context、Skill 和外部 Agent 执行通道。

当前实践可以概括成下面几个层次。

#### 3.1 agentGroup 是运行时宿主，不是业务代码本体

我们把 Harness 和业务项目做了分层：

- `agentGroup` 负责 Persona 管理、Session 编排、上下文拼装、SSE 事件流、Template、Capabilities、Provider 路由。
- 具体项目上下文通过 `AGENTS.md`、`SKILL.md`、Memory 等方式注入。
- 外部执行通道通过 ACP / adapter 接到实际 Agent，例如 `cursor-cli`。

这意味着我们没有把 Harness 写死成“只懂量化策略代码”的系统，而是让它成为一个可承载业务上下文的宿主层。

#### 3.2 我们不是直接把模型接到工具，而是先做上下文工程

这个项目的一个关键判断是：模型不会天然理解业务项目，业务项目上下文必须被工程化注入。

因此我们在 Harness 里显式构造 system prompt，按层拼装：

- 项目级上下文，例如 `AGENTS.md`。
- Persona 的 soul、identity、experience。
- Session 中的 role。
- Template 的补充规则。
- 可用 Skills 的操作手册。

这和“给模型一个 system prompt，然后让它自己想办法”是两种完全不同的路径。我们的做法更像把业务操作规范编译进运行时，而不是把希望寄托在模型临场发挥上。

#### 3.3 我们把多 Agent 协作建模成 Session + Topology

在量化研究里，很多工作并不是一个 Agent 一步做完，而是多个角色在不同结构下协作完成。这个项目把这种协作抽象成统一的 `Session` 和 `Topology`：

- `pipeline` 适合串行接力。
- `broadcast` 适合独立评估。
- `discussion` 适合多轮碰撞。
- `team_task` 适合认领式协作。
- `composite` 适合阶段性编排。

这个抽象很关键，因为它让“因子评审”“红蓝对抗”“论文讨论”“风险复盘”不再是互相独立的硬编码流程，而是同一个运行时上的不同编排配置。

#### 3.4 Persona 不是装饰，而是组织知识与职责边界的容器

在我们的实践里，Persona 不是为了让 Agent 更像真人，而是为了稳定角色分工。

每个 Persona 绑定的是一套长期存在的专业偏好、反模式、经验记忆和默认模型倾向。例如：

- 有的人偏向因子构造与市场微观结构。
- 有的人偏向审计、怀疑和反驳。
- 有的人偏向 regime、风险和环境解释。
- 有的人偏向工程可行性与执行闭环。

这样做的价值在于，协作不是临时拼凑几个“请你扮演某某专家”的 prompt，而是建立可复用的角色系统。Harness 负责把这些角色放入合适的 Session 结构中运行。

#### 3.5 Skill 不是提示词片段，而是操作规范

我们在项目里把 Skill 视为可注入的执行手册，而不是零散 prompt 模板。

Skill 的作用是告诉 Agent：

- 什么时候应该做这类动作。
- 做这类动作时应该遵守什么步骤。
- 使用什么项目内约定、接口、验证方式和输出结构。

这让 Harness 不只是给模型一个任务，而是为任务提供了方法约束。对于量化系统来说，这一点尤其重要，因为研究流程不是自由作文，而是有证据链、验证链和审计要求的。

#### 3.6 我们重视运行隔离、过程可见和结果可追踪

在工程实现上，这个项目把 Harness 做成了一个可观察的运行时，而不是一个黑盒聊天窗口。

例如：

- Session 运行过程通过 SSE 持续输出。
- 事件流里区分 `thinking`、`tool_call`、`tool_result`、`message_complete`、`completed` 等类型。
- 会话有活动流、统计、状态、终止条件和上下文窗口控制。
- 外部系统可以通过 webhook、inject、飞书绑定等方式接入。

这背后的工程取向很明确：我们要的不只是“生成答案”，而是“让一次执行成为系统事件”。

### 4. 我们的实践与通用 Harness 的区别

如果把市场上常见的通用 Harness 理解为“给模型配工具、让它能干活”，那么我们的区别主要体现在下面几件事上。

#### 4.1 我们是业务底座驱动，不是通用能力驱动

很多通用 Harness 的出发点是提升模型通用任务执行能力，比如写代码、查资料、调 API、操作文件。我们的出发点则是量化交易系统中的研究协作链路。

所以我们优先设计的不是通用工具箱，而是：

- 研究角色如何稳定协作。
- 因子工作流如何复用。
- 讨论与执行如何切换。
- 业务上下文如何长期沉淀。

#### 4.2 我们强调“结构化协作”，不是“单 Agent 最大化”

通用 Harness 往往默认一个 Agent 尽可能做更多事。我们的实践更强调：

- 通过 Persona 做角色拆分。
- 通过 Topology 做协作结构建模。
- 通过 Template 固化常见场景。
- 通过 Session 保持过程状态。

也就是说，我们追求的不是一个超级 Agent，而是一套稳定的协作机制。

#### 4.3 我们强调项目上下文注入，而不是默认依赖模型记忆

许多系统假设 Agent 进入仓库后，自然会理解项目结构。我们的实践并不做这种假设。

我们明确认为：

- `AGENTS.md` 是项目上下文入口。
- `SKILL.md` 是操作规范入口。
- Memory 是经验入口。
- Harness 必须主动读取、拼装、注入这些内容。

这是一种更偏工程保守的设计。它牺牲了一部分“看起来很智能的自由度”，换来了更强的稳定性和一致性。

#### 4.4 我们强调“宿主层 + 业务层”分离

在很多场景里，Harness 和业务逻辑会混在一起，最后很难迁移、很难演进。我们的做法是把两层拆开：

- `agentGroup` 负责运行时能力。
- 量化项目负责业务知识、规则、技能和资产。

这样一来，Harness 可以持续演进，而业务项目也可以独立积累自己的方法论与上下文资产。

#### 4.5 我们把 Harness 看成产品基础设施，而不是实验脚手架

实验型 Harness 常常服务于“先跑起来”。而我们的目标是让它逐步成为产品级基础设施。

因此我们在架构上更重视：

- 可配置的 Provider 和模型路由。
- 持久化的 Persona / Template / Session / Activity。
- 前后端一体的控制台与 API。
- 对外集成能力，例如飞书、Webhook、外部注入。

这说明我们的 Harness 不是一次性实验，而是在向长期可运维的系统演进。

### 5. 一个更准确的定位

如果用一句话概括，我们这个项目里的 `Harness Engineer` 可以这样理解：

`Harness Engineer` 负责把量化交易系统中的角色、流程、规范、上下文和执行通道，组织成一套可运行的 Agent 协作底座。

这里的重点不在“模型是否足够聪明”，而在“系统是否能稳定地把聪明组织起来”。

这也是我们和很多通用 Agent 项目的本质区别。我们做的不是一个泛化助手外壳，而是一个面向量化交易场景的、多 Agent、可编排、可观测、可注入业务知识的 Harness。
